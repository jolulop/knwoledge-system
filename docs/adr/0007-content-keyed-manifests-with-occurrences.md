# Content-keyed manifests with occurrences

Each raw source is identified by a content-derived `source_id` of the form `src_<first 16 hex characters of the SHA256>`, and the system writes exactly one manifest per unique content at `raw/manifests/<source_id>.json`. Byte-identical files collapse to a single manifest: every observed path is recorded as an entry in an `occurrences[]` array rather than as a separate manifest. This keeps repeated intake scans idempotent — a rescan preserves the original `discovered_at`, merges any new occurrences, and refreshes only `last_seen_at` and `last_scanned_at`.

Consequences: the earlier per-file `duplicate_of` field is removed from the manifest schema, because a content-keyed manifest would only ever point at itself. Exact (SHA256) duplicates are represented as additional occurrences and, per the human-review policy, do not require approval. Manifest-level `ingestion_status` is normally `new` (or `error`); the "duplicate" disposition is a property of a scan occurrence and the run summary, not of a manifest. Files entering through `raw/inbox/` are assigned `retention_class = unknown` at intake, with retention classification deferred to a later phase. This decision is hard to reverse once manifests exist on disk, so it is recorded here.
