# Context

This document defines the shared language for the Knowledge System project.

## Core Terms

**Raw Source**
An original document, transcript, image, screenshot, HTML file, PDF, DOCX, spreadsheet, or Markdown file stored as the source of truth. Raw sources are not modified by agents.

**Source ID**
A deterministic identifier for a raw source derived from its content: `src_<first 16 hex characters of the SHA256>`. Because it is content-derived, byte-identical files share a single Source ID, which makes repeated intake scans idempotent.

**Manifest**
A JSON record at `raw/manifests/<source_id>.json` describing one unique raw source. There is one manifest per unique content, not per file. It holds the checksum, canonical file metadata, and an `occurrences[]` list. In Phase 1 and Phase 2 the manifest files are the authoritative local runtime listing for the `/sources` endpoint and CLI, but generated `*.json` manifests are not committed to git.

**Occurrence**
One observed file path for a given Source ID. Byte-identical copies of the same content are recorded as additional occurrences inside the one manifest rather than as separate manifests. A redundant copy is an exact (SHA256) duplicate and does not require human review.

**Raw Path**
The manifest may retain an absolute `raw_path` for local runtime operations where resolving the exact file matters. Portable references, API responses, wiki pages, and normalized artifacts should use repository-relative paths such as `relative_raw_path` and `occurrences[].relative_path`.

**Normalized Document**
A parsed representation of a raw source, usually Markdown or JSON, used for indexing, chunking, citation anchoring, and downstream processing. Normalized artifacts are content-keyed: each raw source's Markdown, chunks, tables, and extraction log live in per-source files named by its Source ID, mirroring the manifest model.

**Chunk**
A contiguous span of a normalized document produced by heading-aware splitting, identified as `<source_id>::<ordinal>` and stored per source at `normalized/chunks/<source_id>.jsonl`. Each chunk carries citation anchors so downstream answers can cite verifiable evidence.

**Citation Anchor**
A mechanically derived pointer to where evidence lives in a source — heading/section path and character range (always), source page number (for paginated formats), or table/sheet reference. Anchors are drawn only from the accepted anchors in `policies/citation.yaml` and are never estimated or invented.

**Structured Citation**
The typed object a claim, synthesis, or query answer uses to cite evidence (ADR-0020). Its authoritative locator is `(source_id, char_start, char_end)` — a character range into the source's normalized Markdown — with optional `page`/`page_end`, `section`, and `table_reference`/`sheet_reference`, plus an advisory, non-authoritative `chunk_id`. Free-text "location" strings are not citations.

**Node Identifier**
The typed, stable id every wiki node carries: `source_id` (`src_…`, content hash) for sources, and for semantic nodes a creation-time hash frozen in frontmatter — `concept_id` (`cpt_…`), `entity_id` (`ent_…`), `claim_id` (`clm_…`), `synthesis_id` (`syn_…`), `query_id` (`qry_…`). The id is permanent across renames; the human-facing slug/title may change (ADR-0021).

**Extraction Log**
A per-source diagnostic record at `normalized/extraction_logs/<source_id>.json` describing one extraction run: tool and version, character counts, warnings such as `needs_ocr`, and any error or skip reason. The manifest holds the summary extraction status; the extraction log holds the detail.

**Wiki Page**
A derived Markdown page used for human browsing and agent navigation. Wiki pages are generated or maintained from raw sources and normalized documents. The entire `wiki/` layer — including `index.md` and `log.md` — is mutable local runtime data, not code: it is gitignored, regenerated by the pipeline, and made durable by the backup mechanism rather than by git (ADR-0014).

**Source Page**
A wiki page representing one raw source, stored content-keyed at `wiki/Sources/<source_id>.md` with the human-readable title carried in frontmatter (ADR-0015). In the Phase 3 deterministic backbone a Source page asserts only mechanically-derived facts; sections that require an LLM (tags, concepts, entities, claims, etc.) are rendered as `_Pending semantic enrichment_` placeholders until a later phase fills them (ADR-0013, ADR-0016).

**Input Fingerprint**
A hash recorded in a generated page's frontmatter (`input_fingerprint`) covering every input that determines the page — for deterministic backbone pages: schema version, template, the source's normalized Markdown, manifest fields, and config; for enriched pages it also covers the prompt/template version and the resolved `model_ref` (provider + model id). Regeneration skips a page when the candidate's fingerprint matches the stored one, so template/config/extractor/model changes are picked up without `--force`. Deterministic artifacts (Source pages, `index.md`) carry no wall-clock timestamp; freshness lives in the fingerprint, file mtime, `log.md`, and the job record (ADR-0023, ADR-0027).

**Enrichment**
The Phase 3.5 LLM stage that fills the semantic layer the deterministic backbone leaves as placeholders: real summaries, tags, concepts, entities, claims, and synthesis. Enrichment treats source text as untrusted data and the model as a pure text→data function with no acting tools; its output is schema-constrained, citation-grounded, and human-reviewed for semantic/destructive changes (ADR-0025, ADR-0026). It is provider-agnostic — any model reachable through an LLM Adapter can serve a pass. Enriched pages carry `generation_status: enriched`.

**Enrichment Artifact**
A per-source, content-keyed record of an enrichment pass's validated, schema-conformant output (e.g. `normalized/enrichment/<source_id>.json`). It is the artifact of record for LLM output; enrichment passes write it, never `wiki/Sources/<source_id>.md` directly. The deterministic Source-page renderer stays the sole writer of the `.md` and composes the artifact into the page, filling the `_Pending semantic enrichment._` placeholders when present and falling back to them when absent — so the page remains a pure function of its inputs and a full rewrite never clobbers LLM content. The artifact carries its own `input_fingerprint` (normalized Markdown + prompt/template version + `model_ref` + schema) that drives re-enrichment (ADR-0025, ADR-0027). It is derived state — regenerable by replaying the [[response-cache]] (free, no provider call) when the normalized text reproduces — and is not itself backed up; the cache is the durable record of LLM output (ADR-0027).

**LLM Adapter**
The thin, provider-agnostic seam that abstracts model connection and usage behind one uniform call: `LLMClient.parse(messages, schema, model_ref) -> validated object`, returning a schema-valid object or raising a typed error. Per-provider adapters implement it — native `anthropic` SDK, native `openai` SDK, and a local adapter over the OpenAI-compatible HTTP wire (Ollama, vLLM). Provider-specific accelerators (prompt caching, Batch API) are optional adapter capabilities advertised by flags like `supports_batch`, never guarantees the caller depends on (ADR-0025).

**Model Reference**
The provider-aware identifier a tier maps to, written `provider:model_id` (e.g. `anthropic:claude-haiku-4-5`, `openai:…`, `local:…`). The provider half selects the LLM Adapter and its connection config (base URL, API-key env var); a `local` reference additionally needs a base URL. `model_ref` is what the response-cache key and `input_fingerprint` record, so switching provider or model is a cache miss and a refresh, not a silent collision (ADR-0025, ADR-0027).

**Model Tier**
The configurable model routing used by enrichment: tier 1 (light, default `anthropic:claude-haiku-4-5`) for summaries/tags, tier 2 (standard, default `anthropic:claude-sonnet-4-6`) for concept/entity/claim extraction, tier 3 (heavy, default `anthropic:claude-opus-4-8`) for synthesis/contradiction. Each pass declares its tier in code; only the tier→`model_ref` mapping is config, and it may point all tiers at one model or mix providers (ADR-0025).

**Response Cache**
A persistent local cache of raw LLM responses keyed by `hash(prompt + model_ref + schema)` — `model_ref` includes the provider so the same logical model served via two providers does not collide — stored under `db/` and covered by backups. Because LLM output is not byte-reproducible, the cache lets a rebuild or forced re-render replay the stored response — reproducible and free — until an input changes or the cache is busted (ADR-0027).

**Summary Stub**
A deterministic, extractive `> [!summary]` callout written without an LLM — typically the first meaningful paragraph of the normalized Markdown (or a structural fallback when text is too sparse). It is marked `summary_status: stub` in frontmatter; a later phase replaces it with an LLM summary and flips the flag to `enriched`. The maintenance linter treats a stub as expected, not as summary rot (ADR-0016).

**Concept Page**
A wiki page representing a durable recurring idea. Concept and Entity pages are slug-keyed by their canonical name (e.g. `wiki/Concepts/post-merger-integration.md`) but also carry a stable `concept_id`/`entity_id` in frontmatter that the graph keys edges on, plus an `aliases` list for synonyms — so renames/merges rewrite slugs and wikilinks without breaking edges (ADR-0017).

**Candidate Node**
A promotable node — a concept or an entity-family node (entity/person/organization/project) — evidenced by too few independent sources to be first-class. It is created as a `candidate`/`stub` (low confidence) and kept out of promoted navigation and synthesis. It promotes to `active` automatically once ≥2 *independent* sources evidence it (recurrence), or early by human review; both paths run through the single `promote_candidate_node` review action and the deterministic `promote` worker (ADR-0018, Phase 3.5b slice 5).

**Independent Source**
A source that counts toward a node's promotion threshold. Exact (SHA256) duplicates share one `source_id` and count once. Independence is judged from manifest provenance (`author`/`publisher`/`report_family`/`canonical_url`): two sources are independent iff ≥1 *comparable* key (present on both) differs and no comparable key is equal, after canonicalization (whitespace/case; URLs also drop fragment + trailing slash). Manifests carry no provenance by default, so the gate is conservative — nothing auto-promotes until provenance is populated (ADR-0018).

**Claim**
An atomic factual statement derived from one or more sources. A claim cites its evidence as a Source page plus a stable mechanical anchor (page / section / character range into the normalized Markdown), never a chunk ordinal; a `chunk_id` may be kept only as a non-authoritative pointer (ADR-0019). Claims require citations where possible.

**Synthesis**
A higher-level explanation, comparison, or conclusion across multiple sources, concepts, or claims, stored as a `synthesis` node (`syn_…`) page. In Phase 3.5c it is produced one-per-`active`-concept/entity, eligible only when ≥2 `active` claims connect to the target through the graph neighborhood and those claims are supported by ≥2 *independent* sources (ADR-0018/0031); candidate or single-source targets get no synthesis. It is grounded by referencing already-grounded `claim` nodes (`derived_from` synthesis→claim), with any direct source quote run through the verbatim citation gate (unlocatable → dropped, or generation marked `partial`); those `derived_from` edges are `active` as **provenance, not approval**, so the node itself stays `candidate`. A synthesis page is written to disk under `wiki/Synthesis/` so a human can read it; like a candidate concept it **is listed in `index.md` marked `candidate`** (the index is the full page listing for discovery/review, not a promoted-only view), but is **excluded from promoted/authoritative use** — never returned as evidence for any later synthesis or query answer, and never surfaced by Phase 4 retrieval as evidence — **until promoted to `active`**. The load-bearing invariant is *evidence-eligibility, not index presence*: `candidate` (and `deprecated_candidate`) means readable and navigable but not citable. Its promotion is **review-only via a `propose_synthesis` review item — there is no recurrence auto-promote path**, precisely because a synthesis is born from ≥2 sources by construction and would otherwise auto-promote and bypass review; approval flips it to `active` (mirror + audit), rejection sets `review_status: rejected` and `status: deprecated_candidate` (the lifecycle vocabulary has no `rejected` node status) (ADR-0022, ADR-0028, ADR-0031).

**Contradiction**
A review-gated `contradicts` assertion between two `claim` nodes that conflict, proposed by the Phase 3.5c tier-3 pass. It is written as one edge with the two claim ids **sorted lexically** (`src_id < dst_id`) so the symmetric relation collapses to a single row and a single `resolve_contradiction` review; born `status: proposed` (invisible until reviewed). The edge's evidence anchor (the src claim's primary active citation) is **advisory only** — the authoritative, two-sided evidence stays the two Claim pages' structured citations, and the review item carries both claim ids plus rendered context for both sides. Contradictory claims stay visible until reviewed (Build Spec). A human resolves it with one of three outcomes: `acknowledge` (edge → `active`, both claims live), `supersede` (a named winner gets an `active` `supersedes` edge and the loser is deprecated to `deprecated_candidate` via the `deprecate_wiki_page` audit path, with the cause recorded in `audit_log`; the `contradicts` edge still activates), or `reject` (edge → `rejected`). A `supersede` decision is never silently recorded without its effects. The LLM proposes; the human disposes (ADR-0031).

**Candidate Pair (Blocking)**
The bounded set of claim pairs the Phase 3.5c contradiction pass actually sends to the heavy model, produced by a deterministic, zero-cost *blocking* step over the graph instead of infeasible all-pairs comparison: two claims are a candidate pair iff they share ≥1 `active` concept/entity node (co-mention via `active` `mentions` edges) **and** come from two independent sources (the ADR-0018 independence test — a source cannot contradict itself). Each pair is fingerprint-idempotent and cache-replayed, the fingerprint covering **both claims' text and their active citation anchors plus the shared blocking node ids** (not text alone — evidence context can change while text survives) and the prompt/model/schema; a corpus-level fingerprint is rejected. Vector-similarity blocking is a recorded but deferred second channel (ADR-0031).

**Review Item**
A proposed semantic or destructive change requiring human approval, such as deletion, entity merging, contradiction resolution, deprecation, or archiving.

**Graph (Source of Truth)**
The SQLite store under `db/graph.sqlite` that is authoritative for **relationships (edges)** — keyed on stable typed ids (`source_id`/`concept_id`/`entity_id`/`claim_id`), not slugs, from the governed Build Spec §6.2 vocabulary (minus `needs_review`, which is represented as `status`). Each row is one *assertion* of a relationship — distinct evidence spans and asserters coexist, and a relationship exists/projects iff it has an `active` assertion; each carries `status` (proposed|active|rejected|superseded), provenance (`asserted_by`), confidence, an evidence anchor, and a `review_id`. Node *metadata* (id/title/slug/status/aliases) stays owned by wiki frontmatter (ADR-0017/0021/0022) — sources by their manifest (ADR-0008); the graph's `nodes` table is a derived index rebuilt from those, and the backlink sections in pages are a derived projection of the graph's `active` assertions, so rename/merge is an id-level redirect and backlinks stay synchronized by construction. An LLM- or prose-authored assertion enters as a review-gated `proposed` candidate (pending or deferred stays invisible; approved → active, rejected → rejected) and never appears as a backlink until approved (ADR-0029, ADR-0030; lands in Phase 3.5b).

**Hybrid Retrieval**
The combination of keyword search, semantic/vector search, graph traversal, and summary-first wiki navigation. The two *chunk-evidence* channels (keyword + vector) are fused into one ranked `evidence[]` by [[reciprocal-rank-fusion]]; graph and navigation stay separate `/search` result groups (ADR-0032 decision 6).

**Reciprocal Rank Fusion (RRF)**
The rank-based, scale-free way `/search` combines the keyword (BM25) and vector (distance) chunk-evidence channels into one ranked list (Phase 4e): each chunk scores `Σ 1/(k + rank_c)` over the channels that returned it, `k = rrf_k` (default 60, `policies/retrieval.yaml`). Deduped by the citation `(source_id, char_start, char_end)`; a hit found by both channels merges into one evidence object whose top-level `score` is the fused RRF value, `retrieval_path` lists the contributing channels, and an additive `channels` field carries each channel's 1-based `rank` + native `score` (BM25; vector distance, lower=better). Deterministic: identical query+index ⇒ identical order, tie-break `(source_id, ordinal, char_start)`. Weighted/score-normalized fusion and graph boosts stay **deferred and eval-gated** — their parameters can't be justified without a relevance oracle, whose prerequisite is the [[retrieval-relevance-eval]] corpus (ADR-0032 addenda 7 + 9, ADR-0038); RRF fuses chunk evidence only.

**Structural vs relevance retrieval eval** (ADR-0032 addendum 8, ADR-0038)
Two distinct retrieval evals, never conflated. The **structural retrieval eval** (`tests/test_retrieval_evals.py` + `evals/golden_retrieval.yaml`) runs over a programmatic fixture with a deterministic **fake embedder** and asserts *structure* — anchors, status-awareness, router taxonomy, RRF shape/order, retention. It is **key-free and the CI gate**. The **retrieval relevance eval** (ADR-0038: `evals/corpus/` + `evals/golden_retrieval_relevance.yaml` + an opt-in `scripts/eval_retrieval.py`) measures *semantic relevance* — does `run_search` surface the right citable chunks/sources for a query, by source-level binary judgments scored as recall@k / MRR / success@k. It needs the **real embedder** (no LLM key), so it is **opt-in/manual, never a CI gate**, and produces a reproducibility-stamped report (`embedding_model_ref`, index version, `rrf_k`, `graph_mode=empty`). It is the prerequisite that unblocks weighted RRF / graph-boost tuning, and is distinct from the deferred *answer-quality* eval (`/evals/run`, ADR-0036). v1 cases are **source-level** (relevant/irrelevant by filename); the **multi-chunk extension** (ADR-0038 §Multi-chunk) adds **chunk-level** cases (`category: chunk_disambiguation`) that locate a chunk by a `contains:` phrase resolved to its authoritative **citation key** `(source_id, char_start, char_end)` (not the advisory `chunk_id`), with an **intra-doc `near_miss`** chunk — testing chunk-level *discrimination* (the right chunk ranks above the near-miss) and a **chunk-granular per-channel diagnostic**. Chunk cases are reported separately (`## Chunk-Level Aggregate` / `## Chunk Source Continuity`) and never contaminate the source-level headline; multi-chunk docs use `##` sections to force deterministic chunk boundaries. A chunk failure labelled `keyword_prefers_relevant_vector_prefers_irrelevant` would be the first real evidence that *fusion weighting* (not just a better embedder) could help.

**Answer-quality eval** (ADR-0042)
The third eval, never conflated with the two retrieval evals above: a **key-required, loopback-only, read-only operator workflow** that measures the **cited-answer pipeline** (`POST /query`) against a curated **real-vault golden Q&A corpus** and scores it **deterministically — no LLM judge**. The LLM key is used only to *generate* answers; scoring is mechanical against per-question predicates (expected sources cited, forbidden sources not cited, abstain expected-vs-actual, `unsourced_count`/`security_rejected_count` zero, answer-when-expected, citation recall/precision over `source_id`s). A reusable core `app/workers/eval_answers.py` (fake-client testable, so **CI stays key-free**) backs two thin endpoints: **`POST /evals/run`** (explicit `confirm_cost`, `EVAL_MAX_QUESTIONS_DEFAULT` + hard-cap-clamped `limit`, 503 if the LLM is unconfigured, a key-free `dry_run` corpus pre-flight) and **`GET /evals/results`** (key-free read of stored runs). It is **read-only with respect to vault source-of-truth + semantic state** (writes nothing under `raw/`/`normalized/`/`wiki/`/`reviews/`/`raw/manifests/`/graph; always `/query` with `save:false`), while it MAY write its own result artifacts and MAY populate the LLM [[response-cache]] on a miss like the normal `/query` path (`db/llm_cache.sqlite` is an execution artifact, not vault SoT); `fresh:true` bypasses cache lookup **and** write for a clean provider-drift run, and `cache_hits`/`cache_misses` are recorded. Durable results under `evals/reports/answers/` (gitignored) store **only ids/flags/scores/metadata — no prompt, evidence, or answer prose, and no absolute paths** (privacy: evidence packs + answer text can echo private source text; a root *fingerprint*, not `/home/...`). The real corpus `evals/golden_answers.local.yaml` is **gitignored** (operator data); a committed `evals/golden_answers.example.yaml` documents the schema. The headline is a **reproducibility-stamped snapshot, not a CI gate** (answer generation is nondeterministic). An LLM-as-judge prose-grading "analysis lane" is deferred. Distinct from the deterministic `/query` *structural* eval (`golden_questions.yaml`, fake adapter) and the [[structural-vs-relevance-retrieval-eval]] retrieval evals.

**Cited Answer (Query Answer)**
The output of `POST /query` (Phase 5): an LLM-synthesized answer over [[reciprocal-rank-fusion]] chunk evidence in which **every asserted factual claim is mechanically grounded**. The LLM returns ordered claims that reference retrieved evidence by id only (never inventing anchors); the harness builds each citation from the *retrieved* evidence and runs the verbatim [[citation-grounding]] gate (`ground_citation`, ADR-0026). Only grounded claims enter the answer body — `max_answer_unsourced_claims: 0` applies to the asserted body; ungrounded model output is logged in an audit "Unsourced Claims" section, not asserted. Zero grounded claims → the answer abstains with `"No source found in vault."` Citations come **only** from citable source chunks, never `candidate` node prose (ADR-0032 eligibility invariant). Saved to `wiki/Queries/<query_id>.md` only on explicit request; a saved query is a navigable answer artifact (`type: query`, `answer_eligible: false`) with citations in frontmatter and **no graph edges/authority** (ADR-0034).

**Evidence Pack**
The delimited, **untrusted** bundle of retrieved citable chunks passed to the Phase 5 synthesis prompt: each chunk gets a stable in-request `evidence_id` plus its authoritative anchor (`source_id`, `char_start`, `char_end`) and text/quote. The LLM may reference these only by `evidence_id`; the harness — not the model — builds the citation objects from the pack, so an anchor can never be hallucinated or computed by the model (ADR-0026/0034).

**Review Ledger / Human Review UI**
The Phase 6 governance surface over the `reviews/` artifacts (ADR-0018/0035). **Decide and apply are decoupled:** recording a human decision (approve/reject/defer) only moves the item between `reviews/pending|approved|rejected/` + writes `audit_log/` (`resolve_review_item`), while the *effect* is applied later by deterministic, **key-free**, idempotent executor passes that scan `approved/` (`apply_resolved_syntheses`, `promote_candidates`, `apply_resolved_contradictions`, and the Phase-6 `apply_approved_deprecations`). The **ledger is type-complete** (every pending type is listed + decidable — the human must see everything needing a decision), but `POST /reviews/apply` only runs the executor-backed actions and reports unapplied types/reasons honestly. The UI is **server-rendered HTML on the existing FastAPI app** over a deterministic JSON read model (`GET /reviews`), **never authority** (it calls the same review-service primitives), with mandatory proposal preview before approve; safety rests on the loopback-only no-auth bind (ADR-0009).

**Governance executor risk classification** (ADR-0041)
The durable axis that gates which record-only governance review types get a deterministic executor now vs. deferred: **whether the executor rewrites/collapses/spawns/redirects a stable node id** (`concept_id`/`entity_id`/…, with its backlinks, citations, derived edges, and review subjects). **Non-rekeying** executors (`mark_semantic_duplicate`, `hide_content`, and `change_entity_subtype` *only if* it preserves the id) merely add an edge or flip a status/metadata field on existing ids → buildable now, one small slice each. **Rekeying** executors (`merge_entities`, `merge_concepts`, `split_entity`) collapse or spawn ids and must redirect every reference → highest blast radius, **deferred to a dedicated identity-surgery ADR**. Reversibility and retrieval-visibility are secondary axes, not the gate (the system already has status/eligibility for visibility; rekeying cuts deeper because it changes what ids *mean*, ADR-0021). **`mark_semantic_duplicate`** is the first executor (ADR-0041): on approve it upserts a single **active, canonical, symmetric `duplicates(min_id, max_id)`** assertion (`asserted_by: human`) between two same-type nodes and re-renders a **body-only** `## Duplicates` backlink section on both pages (projected for any active edge regardless of either node's lifecycle status, like all graph projections; human-navigation only, confers no answer-eligibility) — **pure annotation**: no id rewrite, no node-status change, no retrieval suppression, no backlink redirect. Canonical ordering + the wiki↔graph projection are validator-enforced (`validate_graph` rejects a reversed `duplicates` row; `validate_projection` gains both-direction `duplicates` coverage). Subject is the canonicalized pair `{node_ids: sorted([a,b])}`; the executor scope-guard-skips a malformed/invalid/self/cross-type/missing pair (never partial-applies); reject is a ledger-only no-op; it is graph-required and previewable via the [[apply-dry-run]] (edges_added + two wiki diffs; apply does not move approved review files, so no `diff.reviews` entry). A re-apply of an already-projected duplicate is a true no-op (applied/normalized/no-op semantics).

**Hidden status / `hide_content`** (ADR-0043)
The second non-rekeying governance executor: an approved **`hide_content`** review suppresses a **source** from default retrieval + navigation via a new reversible **`hidden`** lifecycle status — **no id rewrite, no deletion; raw + page stay on disk**. It is a near-clone of [[retention-lifecycle]]'s `archive_source`/`apply_archive_sources`: `active → hidden` only (manifest `status` is authority → Source-page mirror → best-effort graph source-node mirror → keyword/nav reindex), with `hidden` added to the status vocabulary but **kept out of `RETENTION_DEFAULT_STATUSES`**, so it is excluded from `/query` evidence + `/search` (incl. the navigation + graph channels) **for free** — like `archive_candidate` (the levers for a *source* are the evidence-retrieval source-status filter + navigation; Source pages are already `answer_eligible: false` regardless of status, so `answer_eligible` is a lever only for the deferred semantic-page slice). `hidden` is **distinct from `archive_candidate`** (governance suppression vs retention staleness; different review types/audit, same visibility mechanism, separately filterable). What "hidden" means is **retrieval + navigation only, NOT graph traversal**: graph APIs still reach a hidden node with its status visible (provenance/reversibility), a backlink may still render on another page (documented v1 limit), and `wiki/index.md` still lists it annotated `hidden`. Scope: **sources only** in v1 (`subject.source_id`; guards mirror `archive_source` — `invalid_source_id`/`source_missing`/`unexpected_to_status`/already-not-active no-op; reject = no-op; executor-backed but **not graph-required**); semantic-node-page hiding and an **unhide** (`hidden → active`) executor are deferred. Previewable via the [[apply-dry-run]] (manifest `active → hidden` diff + Source-page diff).

**Apply Dry-Run / Mutation Preview** (ADR-0040)
The read-only preview of what `POST /reviews/apply` *would* change, served by `POST /reviews/apply/dry-run`. It is **apply-on-a-copy, not an independent projection**: it builds a **fully self-contained** throwaway sandbox root with **no live path reachable from it** — copying the writable domains (`db/` minus `llm_cache`, `reviews/`, `wiki/`, `raw/manifests/`) and the read-only inputs the orchestration touches (`scripts/`, `templates/`, `policies/`, `normalized/`, and the **manifest-referenced** raw bytes so the full validator suite incl. `validate_raw_integrity` runs faithfully) — runs the **same** extracted `run_apply(settings)` orchestration (the 5 executors → `wiki/index.md` rebuild → keyword reindex → validator suite) against the sandbox, then diffs sandbox-vs-live and discards the sandbox — so **executor behavior cannot drift**. Live raw bytes are read-copied into the sandbox but **never modified**; `indexes/`/vector/keyword/`llm_cache` are not copied (regenerated in-sandbox if needed, excluded from the diff). The graph-availability gate lives **inside `run_apply`** so live and dry-run refuse identically (live → HTTP 503; dry-run → a structured `status:"blocked"` / `graph_unavailable` preview). The result is a domain-grouped semantic **diff** (`graph` edge/node-status deltas with stable ids, `wiki` path-scoped unified diffs, `reviews` workflow moves, `manifests` field-level status) plus an `items[]` provenance list (review_id/type/targets → effects), a `not_appliable[]` list (record-only/unsupported types, with reasons — never a fabricated diff), `validators` (would-pass signal; a sandbox validator failure makes it a **failed preview** and the UI withholds Apply), and `warnings[]`. Distinct from the per-type `ReviewPreview` projector (ADR-0035 A1), a single-item UI hint that is *not* the executor path. Dry-run passing is **advisory, not a lock** — live apply still validates after mutation. Sandbox executor failure is a structured failed dry-run with diagnostics, not a 500; a no-op yields an empty diff.

**Unhide / unhide_content + unhide_semantic_page** (ADR-0047)
The governed **inverse** of the hides: a human-approved review item reverses an **already-live** hide (`hidden → active`) via apply — the `EFFECTED`-state effect-reversal ADR-0045 [[reopen-re-decide]] deferred. **Not a reopen:** reopen rewinds a *not-yet-applied decision* (PENDING_APPLY) back to pending; unhide is a *new governance action* against a *live status*. **Separate types** (`unhide_content` source / `unhide_semantic_page` semantic — intent-encoding, not bidirectional hide types); `proposal.to_status: active` is a guard. **v1 scope:** sources + concept/entity-family (claim/synthesis unhide await their hide). **Authority:** source = manifest (`set_status(active)`, reuses `_apply_source_status_transition` generalized with a `from_status` param — hidden→active), best-effort graph mirror, **not graph-required**; semantic = page+graph via `recompose_semantic_node_page(status="active", review_status="none")` — restores the **clean default active state** (uniform with never-hidden pages; governance trail in the ledger/audit), **graph-required**. **Effect:** restores default `/search` nav + graph-channel discovery + `answer_eligible`; reindex-failure → non-clean (`validation_failed` + **`unhide_discovery_restoration_not_guaranteed`** — inverse risk: page active on disk but a stale index still hides it from discovery); `index.md` drops the `hidden` annotation. **Projector/reopen-safety (inverse of [[semantic-page-hiding]]'s `partial_hide_state`):** `EFFECTED` = fully active; `PENDING_APPLY` = still fully hidden (reopenable — no live unhide effect); **partial** (one side active, other hidden) → `UNKNOWN partial_unhide_state` (not reopenable). **Hidden-only guard:** a non-hidden subject → `source_not_hidden`/`node_not_hidden` skip; reject → no-op.

**Semantic-page hiding / hide_semantic_page** (ADR-0046)
The ADR-0043 source-hide model extended to generated **semantic node pages**: an approved `hide_semantic_page` flips a concept/entity-family node **`active → hidden`**, reusing the `deprecate_wiki_page` executor's render seam + guards (`recompose_semantic_node_page(status="hidden")`, subject `{page, node_id}`, canonical-page guard, scope-dir gating). **v1 scope:** concept/entity/person/organization/project (claim + synthesis are fast-follows on their own seams). **Authority:** page frontmatter (authoritative, rendered at `status: hidden` + `review_status: approved`) + graph node (mirror) written together — so it is **graph-REQUIRED** (graph absent / node missing → block + 503; no page-only partial hide), unlike the manifest-authoritative source [[hide-content]]. Stricter reindex posture (mirrors ADR-0043): an applied hide + failed keyword reindex → apply non-clean (`validation_failed` + `semantic_hide_retrieval_suppression_not_guaranteed`); a failed `index.md` *rebuild* stays warning-only. **Effect** (one status flip cascading through existing filters): excluded from default **`/search` navigation** + the **`/search` graph channel** (as adjacent/waypoint) by the `RETENTION_DEFAULT_STATUSES` node-status filter, and its **`answer_eligible` metadata flips false** (needs `status==active`; surfaced on nav rows + `/graph/*` node metadata). `/query` cites **source-chunk evidence** keyed by `source_id` (ADR-0034) — semantic pages are never `/query` citations regardless of status, so hiding changes default `/search` discovery, not `/query`. **Preserved**: the graph node + all its **edges** (graph is SoT — no edge deletion / backlink rewrite) and raw `/graph/*` inspection (still returns it with `status: hidden`). Reuses the `hidden` status (no new status); **active-only, hide-only v1** (non-active → skip/no-op; reject → no-op); **unhide (`hidden → active`) deferred** (a new executor direction, distinct from [[reopen-re-decide]] which reverts a *decision* not a *status*). Reopen + [[apply-dry-run]] integrate free via the `_effect_hide_semantic` projector: `EFFECTED` iff page `hidden` + `review_status: approved` + graph node `hidden`; a **partial live hide** (page XOR graph hidden, or both hidden but review pending) → **UNKNOWN `partial_hide_state`** (NOT reopenable — else [[reopen-re-decide]] would orphan the live hidden page/node); `PENDING_APPLY` only when neither is hidden.

**Reopen / re-decide** (ADR-0045)
A controlled, audited exception to ADR-0035's terminal-decision immutability: a human can **reopen** a terminal review item (move `approved/` or `rejected/` back to `pending/` to be re-decided) **only when its decision's effects are not yet live** — fixing a misclicked [[supersede-via-ui]] `winner` or a wrong approve/reject **without** building an applied-effect undo. The safety oracle is the **existing A2 projector `effect_status`** (no new applied-flag): reopen is allowed only for the statuses that *prove* no live effect — `PENDING_APPLY` (effect verified absent) and `NO_EFFECT_REQUIRED` (owes no world change) — and refused (**409**) for `EFFECTED` (live effect — out-of-scope applied-undo), `UNKNOWN` (can't confirm — repair the read model first), `INVALID_SUBJECT` (tampered), and **`APPLY_DEFERRED`** (`manual_effect_unknown` — a no-executor record-only type means "not applied *by this system*," not "no effect": manual-effect types like `delete_raw_file`/`purge_response_cache` run by hand outside `/reviews/apply`, ADR-0036, and the projector can't detect that; a future ADR may allowlist truly-inert types). Surfaced as a dedicated **`POST /reviews/{id}/reopen`** (graph-aware, unlike the graph-free approve) + a `reviews.reopen_review_item` primitive that clears `decided_*`/`decision_note`/`winner` and appends `audit_log/<id>-reopened-<seq>.json` with `{reason, prior_decision, prior_status, prior_winner?}`; a **non-empty `reason` is required** (400 if blank). The `/ui/reviews/{id}` terminal panel shows a **Reopen** button + reason input only when reopenable. A reopened item is back in `pending/`, so apply + the [[apply-dry-run]] naturally exclude it.

**Supersede-via-UI / contradiction winner** (ADR-0044)
The review-UI affordance for resolving a `resolve_contradiction` by picking the **winning** claim (the loser is superseded). It records the choice as an **optional `winner` field on the *approve* decision** — a sub-outcome of an approved item, not a new verb or status: `approve∅` = **acknowledge** (the `contradicts` edge flips active, both claims stand), `approve{winner∈{claim_a,claim_b}}` = **supersede** (the existing `apply_resolved_contradictions` writes an active `supersedes` edge winner→loser + deprecates the loser to `deprecated_candidate`, `contradicts` stays active for the record), `reject` unchanged. The executor is **reused verbatim** (it already keys off `item["winner"]`); the only new code is recording + validating the winner and the UI. **Decision-time guards** (executor unchanged, dry-run is the backstop): type is `resolve_contradiction`, item pending/deferred, `winner` ∈ the canonical subject pair (else 400 — never a silent acknowledge), `winner` on any other type or on `reject`/`defer` → 400, and **both Claim pages exist with frontmatter `status: active`** (the page frontmatter is the node-status authority, ADR-0022/0030 — so the decision endpoint is **graph-free / never 503s**; graph drift is the dry-run/apply's job), else 409. The recorded `winner` is persisted onto the approved item **and into the terminal `audit_log` entry** (conditionally, when present). The HTML detail page renders the two `proposal.sides` + atomic buttons `[Acknowledge][Supersede: A wins][Supersede: B wins][Reject][Defer]`; `/ui/reviews/{id}/decide` translates `supersede_a/b → approve{winner}` (no JS, loopback-only). The winner is **terminal once approved** (ADR-0035 immutability) — a misclick is a documented v1 limit (un-approve/re-decide deferred); previewable via the [[apply-dry-run]] (`supersedes` edge add + loser `active→deprecated_candidate` + Claim-page diffs).

**Embedding Adapter**
The provider-agnostic seam that turns text into vectors behind one uniform call, `EmbeddingClient.embed(texts) -> vectors`, mirroring the [[llm-adapter]] (ADR-0025). The default adapter speaks the OpenAI-compatible `/embeddings` HTTP wire to a **local** server the operator runs (TEI / vLLM / Ollama on the GPU), so the repository owns no GPU runtime and pulls no Torch into the core install. It owns timeout/bounded-retry and **dimension validation**. A cloud provider is reachable through the same seam only as an explicit, security-gated opt-in (`embedding_api_key`), because sending normalized chunks to a cloud API exports source text across the local-first trust boundary (ADR-0026, ADR-0033).

**Vector Index**
The derived **LanceDB** index under `indexes/vector/` that embeds the *same* per-source chunks as the keyword evidence index. Each row carries the embedding plus citation metadata identical to a keyword evidence hit (authoritative anchor `(source_id, char_start, char_end)`), so a vector hit returns the same [[structured-citation]] object and "joins the same `/search` contract without changing the shape." It is gitignored and regenerable (ADR-0032 §7), backed up only on opt-in, and refreshed **explicitly** (`scripts/reindex_vector.py`, never the per-file hook) — the keyword channel stays the always-fresh path while the vector index may drift between reindexes by design (ADR-0033).

**Embedding Identity (Staleness Key)**
The contract that decides when the vector index must be rebuilt. `embedding_model_ref` **is** the embedding identity — the operator pins it precisely and bumps it whenever the served model, quantization, pooling, normalization, or version changes. The **index-level** key (`embedding_model_ref`, `embedding_code_version`, `distance_metric`, `dimension`, `index_schema_version`) and **per-row** key (`source_id`, `chunk_id`, `chunk_fingerprint`, `embedding_model_ref`) drive rebuilds: any index-level mismatch refuses an incremental run and requires `--force`; a chunk-fingerprint-only diff re-embeds just the changed chunks. Bit-identical vectors are **not** required (cosine ranking tolerates GPU float nondeterminism); runtime guards validate the returned dimension and cross-check the server's `model` against the ref (ADR-0033).

**Summary-First Navigation**
The rule that every major wiki page includes a short `> [!summary]` callout so humans and agents can decide whether to open the full page.

**System of Record**
The canonical project repository under WSL at `~/code/knowledge-system`.

**Maintenance Pass**
A Phase 7 deterministic, job-recorded maintenance operation (`/jobs/lint`, `/jobs/reindex`, `/jobs/stale-check`, eval) that **detects and proposes**: it surfaces health problems, appends `wiki/log.md`, and files review items for anything semantic/destructive, acting autonomously only on safe, non-destructive operations (reindex, backup, candidate detection). There is **no scheduler/daemon** — cadence is an OS `cron` recipe the operator opts into (ADR-0036, ADR-0004). "Failing lint" is an *outcome report*, not an aborted run (distinct from a deterministic `validate_*` integrity hard-fail). The LLM/heuristics propose; a human disposes; `/reviews/apply` actions the approved, executor-backed subset.

**Governance decision vs maintenance task**
The boundary that decides whether a lint finding files a **review item** or is **report-only** (ADR-0037). A *governance decision* asks a human to choose semantic truth or approve a destructive/irreversible action (deprecate, archive, contradiction resolution) → it belongs in the `reviews/` ledger. A *maintenance task* is a health defect with a **mechanical, non-decisional remedy** (re-run a producer) → it is report-only (structured findings + `by_check` counts, no ledger entry). Routing maintenance tasks through the ledger would blur the two and is treated as architectural drift.

**Summary rot** ([[lint-quality-heuristics]], ADR-0037)
A report-only `/jobs/lint` finding (severity **low**): an *enriched* Source summary whose enrichment artifact (`normalized/enrichment/<sid>.json`) `input_fingerprint` no longer matches `artifact_fingerprint(current normalized markdown, current configured summary model_ref)` — i.e. the current enrich pass *would regenerate it* (content/model/prompt/schema drift). A `summary_status: stub` or missing artifact is **not** rot (ADR-0016). Source pages only; remediation code `rerun_enrich`. Concept/Entity/Synthesis summary rot is deferred.

**Synthesis rot** (`synthesis_rot`, ADR-0037 decision 6)
A report-only `/jobs/lint` finding (severity **low**, graph-gated): an *active* synthesis whose durable artifact (`normalized/enrichment/<topic_id>.synthesis.json`) `input_fingerprint` no longer matches `synthesis._fingerprint(current topic, enrich_model_heavy)` — i.e. the topic's evidence (claims/citations/disagreements) drifted since approval. This *is* the synthesis producer's existing `stale_active` signal, surfaced by lint key-free via topic-driven enumeration (`eligible_topics`). `subject` is the `synthesis_id`; remediation code `rerun_synthesis`. An active synthesis whose artifact is missing/unreadable → `synthesis_unverifiable` (degraded). **Evidence-gone** (a topic that no longer reconstructs because active evidence collapsed) is *not* rot and *not* degraded — it is a verifiable governance-lifecycle condition owned by the synthesis producer's deprecation flow. **Concept/Entity summary rot is intentionally absent**: those pages are `generation_status: deterministic` graph projections with no LLM-authored summary, already enforced by [[validate_projection]] (a hard validator); a rot heuristic would duplicate it.

**Stale-claim citation** (`stale_claim_citation`, ADR-0037)
A report-only `/jobs/lint` finding (severity **medium**): a claim's cited anchor no longer supports the **stored** quote. For each durable `normalized/enrichment/<sid>.claims.json` citation that exactly matches an **active graph `derived_from` edge** on `(claim_id, source_id, char_start, char_end)`, the stored quote is re-grounded against the current normalized Markdown (`ground_citation(..., require_quote=True)`); failure → stale. It asserts the *anchor* is stale, **not** that the claim is false. Distinct from `uncited_claim` (a claim with no active source). Remediation code `rerun_extract_claims`. Graph-gated.

**Retention Lifecycle**
The reversible status path a source/page moves along under Phase 7 retention (`policies/retention.yaml`, Build Spec §12): `active → stale_candidate / archive_candidate → archived` (and the human-only `delete_candidate → deleted`). **Archive is a reversible status transition** applied to the manifest + Source page + graph node mirror, then reindex — **never a physical move, rewrite, or deletion of raw bytes** (the locked Phase 7 invariant). Archived material **stays indexed** but is excluded from default retrieval/navigation by the existing status filter (`search.RETENTION_DEFAULT_STATUSES`, ADR-0032 §8); an explicit `source_status=archived` still finds it. The reversible transition is the **`archive_source`** review type (renamed from the misleading `archive_raw_file`), executor-backed in Phase 7 (`apply_archive_sources` flips `active → archive_candidate` — the honest v1 terminal status, never `archived`/physically moved); `delete_raw_file` stays **record-only forever** (proposed/logged/decidable, executed only manually). The **durable Source lifecycle authority is `manifest["status"]`** (default `active`, written only by `manifests.set_status`): the Source page reads it and is a pure projection, so status flows `manifest → Source page → keyword nav index → retrieval filter` and survives a wiki regen (it is *not* page-preserved). `retention_class` (policy category) is distinct from `status` (lifecycle/retrieval visibility). **`missing_raw_source`** is the record-only review type governing a broken source record (a catalogued raw file gone missing — a high-severity lint finding, never auto-remediated, since it may be a transient mount/drive failure). **`purge_response_cache`** is a single aggregate, record-only review type (stable subject `{"scope": "response_cache"}`) proposing a manual purge of the LLM [[response-cache]] when it exceeds the policy `cache_ttl_days`/`cache_max_mb` — it has **no executor** (a bulk purge forfeits reproducibility, ADR-0027), and carries only counts/sizes/age, never cached responses or keys.
